THE PROBLEM

What Goes Wrong Without Structure

Why This Matters

When you ask an AI assistant to build an API, it often skips important steps. It might use outdated information from its training data, make assumptions about what you want, or claim tasks are complete without actually verifying them.

These are the 5 most common gaps we've identified - and the problems they cause:

  • Gap 01
    AI Doesn't Use Your Exact Words
    You say one thing, but the AI searches for something different.
    You: "Use Vercel AI Gateway"
    AI searches: "Vercel AI SDK" (wrong!)
    Should search: "Vercel AI Gateway" (your exact words)
  • Gap 02
    No Proof Files Were Actually Changed
    AI says "I updated all the files" but never runs git diff to prove it.
    AI: "Migration complete! All 6 files updated."
    Reality: Only 4 files were changed, 2 were missed.
  • Gap 03
    Skipped Tests Go Uninvestigated
    AI sees "9 tests skipped" and says "that's expected" without checking why.
    Tests might be skipping because they check the wrong environment variables.
  • Gap 04
    Tasks Marked Done Without Verification
    AI declares "complete!" without checking if the code matches your requirements.
    You asked for "single API key" but code still uses 4 separate keys.
  • Gap 05
    Tests Don't Match Production Code
    Production code uses new patterns, but tests still check old patterns.
    Production: AI_GATEWAY_API_KEY
    Tests: OPENAI_API_KEY (outdated!)

THE SOLUTION

Programmatic Enforcement

How We Fix This

Instead of just asking the AI to follow a process (which it might ignore), we use Python hooks that intercept every action and enforce rules.

Think of it like guardrails on a highway. The AI can still drive, but it literally cannot skip important steps - the system blocks it.

[R]
Research First
AI must search for live documentation before writing any code. No more outdated training data.
[I]
Interview Required
AI must ask YOU questions and wait for answers. No more assumptions about what you want.
[V]
Verification Built-In
System checks implementation matches interview. Warns about mismatches before completing.

HOW IT WORKS

The Hook System

What Are Hooks?

Hooks are small Python scripts that run automatically whenever the AI tries to do something. They check conditions and can either allow or block the action.

There are 3 types: PreToolUse (before an action), PostToolUse (after an action), and Stop (when trying to finish).

YOU: "Build me an AI chat API"
|
AI: Tries to write code
|

PreToolUse Hooks (Before Writing Code)

enforce-research.py Must research first
enforce-interview.py Must ask user questions
verify-implementation.py Check term matching

PostToolUse Hooks (After Actions)

track-tool-use.py Log all research activity

Stop Hook (When Finishing)

api-workflow-check.py Verify all phases done + git diff
|
[ ALLOWED ]
All checks pass
[ BLOCKED ]
Missing requirements

REAL EXAMPLE

Creating a Brandfetch API

The Complete Flow

Let's walk through what happens when you run /api-create brandfetch. This shows the full interview-driven workflow, from research to implementation, with structured questions based on real documentation.

1
You Run the Command
You type the command to create a new Brandfetch API endpoint. The workflow immediately begins.
Your Command:
/api-create brandfetch
2
Research Phase (Automatic)
Before asking any questions, the AI is required to research. It fetches live documentation from Context7 and searches the web.
Research Logged to State File:
- Context7: Brandfetch SDK docs → found logos, colors, fonts endpoints
- WebSearch: "Brandfetch API rate limits 2025" → 5 req/second
- WebSearch: "Brandfetch API response format" → JSON with asset URLs
- Context7: Authentication → Bearer token required
3
Structured Interview (Based on Research)
Now the AI asks YOU questions - but with multiple-choice options derived from what it actually found in the documentation. No guessing!
Question 1 (with options from research):
"What's the primary purpose of this endpoint?"

1. Brand lookup (get brand by domain)
2. Logo extraction (get logo assets)
3. Color palette extraction
4. Full brand kit (all assets)
5. Type something else...

You select: "4. Full brand kit (all assets)"
4
More Questions, All Tracked
Each question is tracked with your answer. These decisions are stored in the state file and injected during implementation to ensure consistency.
Decisions Captured:
• Purpose: full_brand_kit
• Response Format: JSON with asset URLs
• API Key Handling: server environment variables only
• Error Handling: detailed (error, code, details)
• Required Params: domain (string)
• Optional Params: include_colors, include_fonts
5
Implementation (With Decision Reminders)
When the AI writes code, the hook injects your decisions as a reminder. This ensures the implementation matches what you actually asked for.
Injected on Every Write:
✅ Interview complete. REMEMBER THE USER'S DECISIONS:

• Purpose: Full brand kit (all assets)
• Response Format: JSON with asset URLs
• API Key Handling: server environment variables only
• Error Handling: detailed (error, code, details)

Your implementation MUST align with these choices.
6
TDD + Final Result
Tests are written first (TDD), then implementation, then docs. Everything is tracked and verified against your interview decisions.
Files Created:
- src/lib/schemas/brandfetch.ts (Zod validation)
- src/lib/__tests__/brandfetch.test.ts (tests first!)
- src/app/api/v2/brandfetch/route.ts
- Interview doc: src/v2/docs/endpoints/brandfetch.md
- All tests passing ✓
